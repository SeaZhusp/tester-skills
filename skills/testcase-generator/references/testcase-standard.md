# 功能测试用例标准

## 概述

本文档定义功能测试用例的生成标准，确保测试覆盖全面、表述清晰、可执行性强。

## 用例结构

### 必填字段

| 字段 | 说明 | 示例 |
|------|------|------|
| 用例编号 | 唯一标识，采用 TC_模块_序号 格式 | TC_LOGIN_001 |
| 所属模块 | 功能所属模块 | 登录模块 |
| 用例标题 | 简洁描述测试点 | 用户名正确，密码正确，登录成功 |
| 用例类型 | 功能测试/边界测试/异常测试 | 功能测试 |
| 优先级 | P0/P1/P2 | P0 |
| 前置条件 | 测试执行前必须满足的条件 | 1. 用户已注册<br>2. 用户已打开登录页面 |
| 测试步骤 | 逐条描述操作步骤 | 1. 输入用户名: test<br>2. 输入密码: 123456<br>3. 点击"登录"按钮 |
| 预期结果 | 每步对应的预期行为 | 1. 输入框显示对应内容<br>2. 密码显示为***<br>3. 页面跳转至首页 |

## 测试覆盖维度

### 1. 正向功能测试

验证核心业务功能的正常流程。

**示例场景**：
- 用户能正常完成注册、登录、登出
- 数据能正确保存和展示
- 业务流程能顺利完成

### 2. 反向功能测试

验证异常情况和错误处理。

**覆盖点**：
- 空输入 / 空字符串
- 超长输入
- 特殊字符（SQL注入、XSS）
- 格式错误（邮箱、手机号等）
- 重复操作（重复提交、重复登录）
- 并发操作

### 3. 边界值测试

测试边界和极限条件。

**边界类型**：
- 数值边界：最小值、最大值、比最小值小1、比最大值大1
- 字符边界：空、单字符、超长（最大长度+1）
- 时间边界：最早时间、最晚时间、过期时间
- 数量边界：0、1、最大数量、最大数量+1

### 4. 等价类划分

将输入数据合理分组，每组选取代表性数据。

**划分原则**：
- 有效等价类：符合规则的数据
- 无效等价类：不符合规则的数据

**示例**（密码验证）：
| 等价类 | 示例数据 | 预期结果 |
|--------|---------|---------|
| 有效-符合规则 | Test123456 | 通过 |
| 无效-太短 | Test12 | 提示密码太短 |
| 无效-太简单 | testtesttest | 提示密码强度不足 |
| 无效-含空格 | Test 1234 | 提示格式错误 |

### 5. 状态与流程测试

验证状态转换和业务流程。

**关注点**：
- 状态定义是否清晰
- 状态转换条件是否正确
- 非法转换是否被正确阻止
- 业务流程是否完整

**示例**（订单状态机）：
- 待支付 → 已支付 → 已发货 → 已收货 → 已完成
- 待支付 → 已取消
- 已发货 → 退货中 → 已退货

### 6. 场景测试

基于用户实际使用场景的端到端测试。

**场景要素**：
- 角色：用户类型
- 目标：用户想完成什么
- 步骤：完成目标的路径
- 预期：期望的结果

**示例**：
| 场景 | 角色 | 步骤 | 预期 |
|------|------|------|------|
| 正常购物流程 | 普通用户 | 浏览→加入购物车→结算→支付→查看订单 | 订单创建成功 |
| 优惠券使用 | 普通用户 | 选择商品→使用优惠券→结算→支付 | 优惠金额正确扣减 |

### 7. 权限测试

验证用户权限控制是否正确。

**覆盖点**：
- 未授权访问：未登录用户访问需授权页面或接口
- 越权操作：低权限用户执行高权限操作（如普通用户尝试删除数据）
- 角色权限：不同角色访问各自权限范围内的功能
- 权限变更：权限变更后即时生效
- 页面级权限：菜单、按钮、链接的可见性控制

**示例**：
| 场景 | 角色 | 操作 | 预期 |
|------|------|------|------|
| 未登录访问 | 游客 | 直接访问用户列表页 | 跳转登录页 |
| 越权操作 | 普通用户 | 尝试访问管理后台 | 提示无权限 |
| 角色权限 | 管理员 | 访问用户管理模块 | 正常访问 |
| 权限变更 | 管理员 | 撤销某用户权限后 | 该用户即时失去权限 |

### 8. 网络异常测试

验证网络异常情况下的系统行为。

**覆盖点**：
- 网络断开：断网后操作不受影响，数据不丢失
- 请求超时：接口请求超时后的错误处理
- 网络切换：WiFi ↔ 移动数据切换时的稳定性
- 弱网环境：加载缓慢、超时重试机制

**示例**：
| 场景 | 操作 | 预期 |
|------|------|------|
| 提交时断网 | 断网后点击提交 | 提示网络异常，数据保留 |
| 请求超时 | 接口超过设定时间未响应 | 提示请求超时，可重试 |
| 弱网加载 | 弱网环境下加载列表 | 显示加载进度或超时提示 |

### 9. 并发测试

验证并发场景下的数据一致性和系统稳定性。

**覆盖点**：
- 重复提交：快速重复点击导致多次提交
- 并发抢购：多人同时抢购有限库存
- 数据竞争：多人同时编辑同一数据
- 状态冲突：并发操作导致状态不一致

**示例**：
| 场景 | 操作 | 预期 |
|------|------|------|
| 重复提交 | 快速点击提交按钮两次 | 只执行一次提交 |
| 并发抢购 | 100人同时抢购1件商品 | 只有1人成功 |
| 数据编辑 | 两人同时编辑同一表单 | 后提交者提示数据已变更 |

## 表述规范

### 测试步骤规范

1. **明确具体**：每步描述一个操作
2. **可执行**：步骤可以直接执行
3. **有预期**：每步有明确的预期结果
4. **独立性**：步骤之间相对独立

### 常见表述模板

**操作类**：
- 点击"XXX"按钮
- 输入"XXX"到XXX输入框
- 选择"XXX"选项
- 勾选/取消勾选"XXX"复选框
- 下拉选择"XXX"
- 长按"XXX"元素

**验证类**：
- 页面跳转到XXX
- 弹出"XXX"提示框
- 显示"XXX"错误信息
- XXX按钮变为可用/不可用
- XXX列表显示N条记录

### 避免事项

1. ❌ 避免模糊描述："输入正确信息"、"点击确定"
2. ❌ 避免跳跃步骤："直接登录"而非"输入账号密码后点击登录"
3. ❌ 避免合并步骤：每个操作单独成步
4. ❌ 避免循环步骤：不要写"重复以上步骤"

## 优先级划分原则

| 优先级 | 适用场景 | 占比建议 |
|-------|---------|---------|
| P0 | 核心功能、基础流程、主流程 | 20-25% |
| P1 | 重要功能、主流场景 | 30-60% |
| P2 | 边缘功能、异常场景  | 10-25% |

## 质量检查清单

生成功能测试用例后，检查以下要点：

- [ ] 每个P0用例都有正向和反向测试
- [ ] 边界值已覆盖
- [ ] 等价类划分合理
- [ ] 测试步骤可执行
- [ ] 预期结果明确
- [ ] 无重复用例
- [ ] 优先级划分合理
